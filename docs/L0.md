# F5.1 实时监控推送系统 架构总纲 (V1.0)

> **系统标识**: F5.1  
> **包名**: stockpush  
> **版本**: 1.0.0  
> **更新日期**: 2026-07-13  
> **用途**: 基于 PostgreSQL 的实时股票信号计算与飞书推送

---

## 0. 核心治理公约

### 0.1 编码规范

- **语言**: Python 3.10+
- **命名规范**:
  - 类名: `PascalCase`（如 `FeishuPusher`, `FunctionEngine`）
  - 函数/方法: `snake_case`（如 `run_headless`, `get_symbols`）
  - 常量: `UPPER_SNAKE_CASE`
  - 模块名: `snake_case.py`
- **类型注解**: 所有函数签名必须标注类型；`Optional` / `List` / `Dict` 优先使用 `typing` 模块
- **导入顺序**: 标准库 → 第三方 → 项目内部；每组之间空行
- **异常处理**: 调用方自行处理异常；服务类不吞没业务异常，仅记录日志后向上传播
- **配置优先**: 运行期参数优先从 `config.yaml` + `config.local.yaml` 读取，环境变量次之，避免硬编码

### 0.2 治理章程

- **模块隔离**: `services/` 下各模块职责单一，禁止跨模块调用非 public API（即非 `_` 前缀的类/函数视为 public）
- **数据库操作**: 统一通过 `PGConnector` 访问 PostgreSQL；禁止模块内直接使用 `psycopg` 原始连接
- **数据源访问**: 统一通过 `SrcMgrService` + `DataSourceRegistry` 接入，禁止模块绕过注册表直接初始化 Provider
- **凭据管理**: 敏感凭据（飞书 Webhook URL、签名密钥）必须通过 `credential_store.py`（Fernet 加密）存储和读取；禁止明文写入配置文件
- **日志**: 统一使用 `logging.getLogger(__name__)` 按模块名命名；禁止 `print()` 输出（`console.py` 交互菜单除外）
- **配置分层**: `config.yaml` 提供默认值，`config.local.yaml` 覆盖运行期配置；运行期配置变更不修改 `config.yaml`

### 0.3 架构目标

1. **实时性**: 交易时段内按固定间隔（默认 60s）完成抓数 → 计算 → 归档 → 推送全流程
2. **可靠性**: systemd 定时器保证开盘启动、收盘停机；数据抓取支持来源切换与完整性检查
3. **可扩展**: 信号策略以用户函数（`userfunc/`）形式动态加载，无需重启即可注册/启用/禁用
4. **可观测**: 启动/停机/除权除息/数据源切换均推送飞书通知，提供 HermesAPI 远程查询状态

### 0.4 数据源调用规范

- **Primary 数据源**: 由 `config.yaml` 的 `datasources.primary` 指定，默认 `xtick`
- **数据源适配**: 每个数据源实现 `BaseProvider` 接口，通过 `DataSourceRegistry` 注册；`SrcMgrService` 统一对外暴露
:- **降级策略**: `DataFetcher` 内置来源切换：primary 失败时自动降级至备用数据源；切换后通过 `FeishuPusher.push_datasource_switch()` 通知
:- **交易日判断**: `CalendarChecker.is_today_trading_day()` 使用 XTick 交易日历接口

### 0.5 完成标志

以下条件同时满足视为一次信号轮次"完成"：

1. 所有启用函数（`FunctionRegistry.get_enabled()`）均已执行
2. 每个函数产生的信号已写入 `tb_signal_log`
3. 信号已通过 `FeishuPusher.push_signal()` 推送到飞书
4. 当天新产生的数据已在 `tb_download_request_log` 记录审计日志

## 1. 系统架构

### 1.0 双架构说明


两个环境的 API 接口兼容（`execute_query` / `execute_update` / `execute_many`），代码层无需条件分支。

### 1.1 业务需求核心要素

| 要素 | 说明 |
|------|------|
| 交易时段 | 上午 09:25–11:35，下午 13:00:05–15:05，间隔 60s 执行一次 |
| 信号标的 | 从 `tb_stock_pool` 读取自选股列表，支持股票与基金 |
| 信号策略 | 用户自定义函数，策略模块位于 `userfunc/`，由 `FunctionRegistry` 运行时动态注册。当前包含：MF30 / MF05 / TF05（旧版 5M/30M 策略）+ mm2 / mm2_i5 / mm2_i5_min1（新版 MM2 I5 通道多周期策略，支持 1m/5m/30m/1d） |
| 数据周期 | 1m / 5m / 30m / 1d 四档 K 线，主力周期为 5m 和 30m |
| 推送渠道 | 飞书 Webhook（支持 HMAC 签名），另提供 HermesAPI 供远程查询 |
| 部署方式 | systemd timer 驱动：`f51-start.timer`（09:00 触发）、`f51-stop.timer`（15:30 触发） |

### 2.2 物理存储契约

F5.1 使用 PostgreSQL 作为持久化存储，共管理 13 张核心表（由 `sys_init.py:F51_TABLE_SQLS` 幂等创建）：

| 表名 | 用途 | 访问模式 | 数据来源 |
|------|------|----------|----------|
| `tb_sys_cfg` | 系统运行期配置 | 读写 | 系统初始化 / HermesAPI |
| `tb_stock_codes` | 股票代码主表 | 只读 | 代码库同步 |
| `tb_fund_codes` | 基金代码主表 | 只读 | 代码库同步 |
| `tb_stock_pool` | 自选股池 | 读写 | PoolMgrService / PoolWatcher |
| `tb_calendar` | 交易日历 | 只读 | XTick 同步 |
| `tb_raw_1m` | 1 分钟 K 线 | 写入（追加） + 读取 | DataFetcher 定时抓取 |
| `tb_raw_5m` | 5 分钟 K 线 | 写入（追加） + 读取 | DataFetcher 定时抓取 |
| `tb_raw_30m` | 30 分钟 K 线 | 写入（追加） + 读取 | DataFetcher 定时抓取 |
| `tb_raw_1d` | 日 K 线 | 写入（追加） + 读取 | DataFetcher 定时抓取 |
| `tb_signal_functions` | 信号函数注册表 | 读写 | FunctionRegistry 管理 |
| `tb_signal_function_params` | 信号函数参数集 | 读写 | FunctionRegistry 管理 |
| `tb_signal_log` | 信号存档 | 写入 + 读取 | FunctionEngine → SignalStore |
| `tb_download_request_log` | 下载请求审计日志 | 仅 INSERT | DownloadLogger 定时记录 |

数据源 Provider 注册表 `tb_datasource_registry` 在 F5.1 中仅做幂等播种，不依赖运行时查询（由 `DataSourceRegistry` 初始化时写入）。

### 2.3 字段标准化

| 规则 | 约定 | 适用范围 |
|------|------|----------|
| 时间戳列名 | `ts` 表示 K 线时间戳；`created_at` / `updated_at` 表示记录创建/更新时间；`signal_time` 表示信号时间 | 全部 `tb_raw_*` 和日志表 |
| 股票代码 | `symbol VARCHAR(10)`，如 `"600000"`，不包含交易所前缀 | 全部含 symbol 的表 |
| 精度 | `DECIMAL(10,3)` 表示股价，`BIGINT` 表示成交量，`DECIMAL(20,2)` 表示成交额 | K 线表 |
| 布尔 | `BOOLEAN` 类型，禁止 `INT 0/1` 表示逻辑值 | 全部含布尔字段的表 |
| 信号方向 | `direction VARCHAR(10)`，取值 `"buy"` / `"sell"` / `"hold"` | `tb_signal_log` |

### 2.4 资产范围与共识校验

- **覆盖资产**: A 股股票 + 场内基金（`tb_stock_pool.type IN ('stock', 'fund')`）
- **自选股权威源**: `tb_stock_pool` 为唯一权威自选股来源；`PoolWatcher` 通过哈希比对检测变更，上层 `symbols_getter` 回调获取最新列表
- **数据无共识校验**: F5.1 采用单数据源方案（primary 自动降级），不要求多源交叉校验；降级切换时通过飞书通知记录

### 3.4 跨模块调用约束

| 调用方向 | 允许范围 | 禁止 |
|----------|----------|------|
| `services/*` → `pg_connector.PGConnector` | 所有服务均依赖 PGConnector | 绕过 PGConnector 直接使用 psycopg |
| `services/*` → `credential_store` | 仅 `worker.py` 和 `HermesAPI` 可调用 | 其他模块不得直接调用加密/解密 |
| `services/*` → `src_mgr/*` 或 `src_mgr/` | 通过 `SrcMgrService` 调用；`DataSourceRegistry` 由 `src_mgr.py` 管理 | 服务模块绕过 `SrcMgrService` 直接实例化 Provider |
| `userfunc/*` → `pg_connector` | 用户函数可通过 `_fetch_data()` 函数内调用 PGConnector 取数 | 用户函数不得直接操作非 `tb_raw_*` 表 |
| `console.py` → `HermesAPI` | Console 所有操作均委托 HermesAPI | Console 不得直接实例化 Service |

## 4. 重置策略（全局）

### 4.1 硬重置（Hard Reset）

适用于数据库损坏、表结构变更后重建：

```
# 1. 清理全部 F5.1 数据
DROP TABLE IF EXISTS
  tb_sys_cfg, tb_stock_codes, tb_fund_codes, tb_stock_pool,
  tb_calendar, tb_raw_1m, tb_raw_5m, tb_raw_30m, tb_raw_1d,
  tb_signal_functions, tb_signal_function_params, tb_signal_log,
  tb_download_request_log, tb_datasource_registry
CASCADE;

# 2. 重新建表
python -m stockpush.services.sys_init

# 3. 重新创建适配器用户函数目录（worker.py 自动创建）
python -m stockpush.worker --headless  # 自动重建 userfunc/__init__.py
```

### 4.2 软重置（Soft Reset）

适用于运行时重置状态，不破坏数据：

```
# 1. 清空信号日志
DELETE FROM tb_signal_log;

# 2. 重置自选股池哈希（触发 PoolWatcher 全量重载）
#    下次 PoolWatcher.check_changes() 自动捕捉差异

# 3. 清除 FunctionRegistry 模块缓存（运行时）
#    通过 HermesAPI 重启进程即可
```

### 4.3 IMMUTABLE 边界

以下数据在任何重置策略中均**不可删除或覆盖**：

| 数据 | 理由 | 保护措施 |
|------|------|----------|
| `config.yaml` | 系统默认配置版本控制 | Git 跟踪，`config.local.yaml` 覆盖模式不修改 `config.yaml` |
| `config/telegram_credentials.key` | Fernet 加密密钥，丢失后无法解密已存凭据 | `credential_store._ensure_key()` 仅创建不覆盖 |
| 用户函数源码（`userfunc/*.py`） | 信号策略自持代码，非系统生成 | Git 跟踪，不纳入重置范围 |
| `tb_signal_log` | 历史信号记录不可回溯覆盖 | `SignalStore.write_signals()` 内置重复检测（func_id + symbol + direction + signal_time），仅追加不覆盖 |

### 4.4 数据库连接断开后的恢复

- `PGConnector._ensure_connection()` 在每次 `execute_query/execute_update/execute_many` 前检查连接状态，断开时自动重连
- 连接恢复后**不丢失数据**：因连接断开而失败的 SQL 由调用方重试（`FeishuPusher.push()` 内置 3 次重试；`DataFetcher` 数据抓取遵循来源降级逻辑）
