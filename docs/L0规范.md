# L0 规范：架构总纲编写规范（V2.0）

> **版本**: V2.0  
> **更新日期**: 2026-07-01  
> **用途**: 约束 `L0.md` 的结构、内容边界与质量门禁  
> **说明**: 本文档自包含，编写 L0 时无需参考其他规范文档

---

## 0. 文档定位

L0 是系统级"架构总纲"，作用等同于"序言 + 目录 + 治理公约"。

- **回答什么问题**：系统有哪些功能模块、各模块边界在哪里、技术与数据治理的硬规则是什么。
- **不回答什么问题**：单个模块怎么实现、UI 控件细节、算法步骤与伪代码、任务拆解（这些属于 L1）。

**唯一性要求**（强制）：
- `L0.md` 为全局唯一架构总纲文档；禁止创建 `L0_v2.md` 等分裂文档。

---

## 1. 标准结构（L0.md 模板）

L0 采用以下固定结构，允许扩展但不得删除核心段落：

```markdown
# <系统名称> 架构总纲 (Vx.y)

## 0. 核心治理公约
### 0.1 编码规范
### 0.2 治理章程
### 0.3 架构目标
### 0.4 数据源调用规范
### 0.5 完成标志

## 1. 系统架构
### 1.0 双架构说明（如有）
### 1.1 业务需求核心要素
### 1.2 系统功能导航表

## 2. 数据契约（全局）
### 2.1 技术栈
### 2.2 物理存储契约
### 2.3 字段标准化
### 2.4 资产范围与共识校验

## 3. 模块依赖图

## 4. 重置策略（全局）
```

## 2. 内容边界

### 2.1 必须包含

1. **核心治理公约**：编码规范、治理章程、架构目标、模块隔离、数据源规范、完成标志。
2. **系统功能导航表**：每个功能 1 行，至少包含：
   - 功能 ID
   - 模块名称
   - 物理路径（Service）
   - 物理数据表
   - 核心职责（1 句话）
3. **全局数据契约**：技术栈、物理存储契约、字段标准化、资产范围。
4. **模块依赖图**：模块间调用关系和数据流方向。
5. **重置策略**：软/硬重置规则、IMMUTABLE 边界。

### 2.2 禁止包含

- 单模块的接口清单、交互细节（属于 L1）
- 单模块数据表字段/索引/约束的完整定义（属于 L1）
- 任何实现步骤、伪代码、类内部方法步骤（属于 L1 接口签名）
- AI Prompt 模板（已废弃，不再使用）
- 任务拆解与测试用例（不再使用 L2）

## 3. 编写规则

### 3.1 功能导航表

- **核心职责**必须 1 句话描述可验证的"模块产出/模块边界"，避免堆砌名词。
- 不出现实现细节，例如"使用线程池""用某算法"等。
- **物理路径**必须写到目录级或关键文件级。
- **数据表**只写"表名列表"，不写字段、索引、约束（属于 L1）。

### 3.2 数据契约

- 字段标准化只定义全局规则（列名、类型、格式），不定义单模块字段列表。
- 物理存储契约只定义"哪些周期/哪些表是物理源"，不定义表结构。

### 3.3 模块依赖图

- 用简洁的文字或 ASCII 图示表达调用关系和数据流方向。
- 不画 UML，不写接口参数。

---

## 4. 质量门禁

L0 文档提交时必须满足以下条件：

1. 功能导航表覆盖所有存在的 Service 模块，无遗漏
2. 每个功能模块有唯一的 Fx.y 编号
3. 数据契约与实际代码中 `pg_connector.py` / `sql/` 目录一致
4. 依赖图与实际 `import` 关系一致（可通过 `ast_grep` 验证）
5. 无"TBD""TODO（忽略此项）"等占位标记
