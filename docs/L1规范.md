# L1 规范：功能契约编写规范（V2.0）

> **版本**: V2.0  
> **更新日期**: 2026-07-01  
> **用途**: 约束 `docs/L1/Fx.y_<功能>_L1.md` 的结构、内容边界与质量门禁  
> **说明**: 本文档自包含，编写 L1 时无需参考 L0 规范或其他文档

---

## 0. 文档定位

L1 是"功能契约"，风格类似 OOP 的 Class 定义：定义**是什么**，不定义**怎么做**。

- **回答什么问题**：模块做什么、接口契约是什么、数据表如何定义、依赖哪些模块。
- **不回答什么问题**：实现步骤、伪代码、具体线程/队列/SQL实现（这些在代码中体现）。

**唯一性要求**（强制）：
- 每个功能只允许一份 L1：`docs/L1/Fx.y_<功能名>_L1.md`
- 禁止创建 `Fx.y_v2.md`、`Fx.y_修订版.md` 等分裂文档

## 2. 内容边界

### 2.1 必须包含

1. **功能概述**：1-2 句话，明确业务价值与范围。
2. **物理定位**：Service 路径、数据表拥有权、配置文件。
3. **核心要素定义**：
   - 关键规则（算法契约、业务约束、边界条件）
   - 数据表结构（字段/类型/约束/索引）← **重点章节**
4. **数据依赖**：拥有并管理 / 只读访问 / 不直接访问 三分区。
5. **接口规范**：← **重点章节**
   - Export API：函数签名 + 类型标注 + 用途说明
   - Import API：依赖谁、调什么
6. **配置文件规范**：路径 + key 结构。
7. **Reset 机制**：软/硬重置、IMMUTABLE 数据边界。
8. **验收标准**：完成标志清单。

### 2.2 禁止包含

- 实现步骤、伪代码、SQL 语句、线程模型细节
- UI 控件清单的详细描述（只写交互规则，不写 HTML 元素清单）
- 任务拆解与角色交替（不再使用 L2）
- 测试用例代码（自动化验证覆盖，不手写）

## 3. 重点章节编写规则

### 3.1 数据表结构（重点）

数据表是系统中最稳定的契约，必须完整定义：

```markdown
### 3.2 数据表结构

#### tb_xxx（模块拥有）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INTEGER | PRIMARY KEY, AUTOINCREMENT | 主键 |
| ts | TIMESTAMP | NOT NULL | 时间戳 |
| symbol | VARCHAR(10) | NOT NULL | 股票代码 |
| ... | ... | ... | ... |

**索引**：
- UNIQUE(ts, symbol)
- INDEX(symbol)

**拥有权**：本模块拥有写入与重置权限。
```

### 3.2 接口规范（重点）

接口签名必须包含类型标注：

```python
def fetch_and_save(symbol: str, period: str) -> Tuple[bool, int, str]:
    """拉取并保存数据"""
```

---

## 4. 质量门禁

L1 文档提交时必须满足：

1. 数据表定义与实际 `sql/` 目录中 DDL 完全一致
2. Export API 签名与实际 Python 函数签名一致
3. Import API 准确列出依赖模块和调用函数
4. 完成标志每一项都是可验证的客观标准
